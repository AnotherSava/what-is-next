# Project guidelines — What's Next

## Local development

- **The local dev server runs strictly on port 3939 — never Next.js's default 3000.** The port lives in `config/deploy.env` (`DEV_PORT=3939`, `DEV_DIR=web`).
- **Launch/restart it with `deploy` (or `bash scripts/deploy.sh`) — either works from any directory.** Both resolve the repo root and read `config/deploy.env`, so the server always comes up on 3939: the `deploy` shell function walks up to the repo root via `run_repo_script`, and the underlying deploy scripts independently resolve the root (nearest ancestor holding `config/deploy.env`, via the shared `_repo-dir.sh` helper). `deploy` is just shorter.
- **Never run a second dev server *for this project* on another port.** Next.js 16 allows only one dev server per project directory: starting a second `next dev` for `web/` (even on a different port) makes the new one defer to the already-running instance, so 3939 never actually comes up even though `deploy` reports "ready" and the port briefly opens. Symptom: `http://localhost:3939` unreachable while a stray what-is-next `next dev` (one started by hand on another port) is alive. Fix: stop the stray instance, then restart 3939. (Dev servers of *other* projects — e.g. printlab on 3000 — don't conflict; the lock is per-project, not machine-wide.) Access the app only at `http://localhost:3939`.
- **Restart the dev server after `prisma generate`.** A running `next dev` keeps a stale in-memory Prisma client, so pages touching a newly added model/field throw `Cannot read properties of undefined` / "unknown field" 500s until restart.
- **Run `npx prisma generate` explicitly after `prisma migrate dev`.** Its built-in generate step didn't refresh the generated client here — `tsc` kept reporting a just-added field as nonexistent until an explicit `prisma generate` (then restart dev, per the note above).
- **`prisma migrate dev` aborts in this non-interactive shell** ("environment is non-interactive, which is not supported" — even with `--name` or `--create-only`). To add a migration, hand-author `prisma/migrations/<YYYYMMDDHHMMSS>_<name>/migration.sql` (for a nullable column + index that's just `ALTER TABLE … ADD COLUMN` + `CREATE … INDEX` — no table rebuild) then apply with `doppler run -- npx prisma migrate deploy`, then `npx prisma generate` and restart dev (per the notes above).
- **`prisma db execute` reads `prisma.config.ts`, not `--schema` (Prisma 7).** Passing `--schema` fails with "unknown or unexpected option: --schema" — drop it. To run one-off raw SQL against the dev DB (e.g. flip a `Setting` flag to A/B a behavior), use `doppler run -- npx prisma db execute --stdin <<'SQL' … SQL`. It executes statements only and prints no query results, so read rows back with a small `PrismaClient` script or Prisma Studio.
- **Deleting an app route can leave a stale `.next/types` type error.** After removing a `page.tsx` route, `npx tsc --noEmit` may fail with `Cannot find module '.../page.js'` from `.next/types/validator.ts` — a leftover from a prior `next build` that dev never regenerates (dev uses `.next/dev/types/`); a dev-server restart doesn't clear it. Fix: `rm -rf .next/types` (regenerates on the next build).
